# Encrypted Agent Memory API — Design Spec

**Date:** 2026-03-14
**Status:** Approved
**Summary:** Evolve Context Chest into an encrypted agent memory platform built on OpenViking, serving both developers (API) and consumers (personal vault) from day one.

---

## 1. Product Vision

**"The AI agents remember. You control who knows what."**

Context Chest becomes an encrypted memory layer for AI agents. OpenViking provides the context database engine (filesystem paradigm, L0/L1/L2 tiers, semantic search, session management). Context Chest wraps it with end-to-end encryption, zero-knowledge auth, consent-based agent connections, and audit logging.

**Positioning:**
- vs Mem0/Zep: "Same memory capabilities, but encrypted and consent-based"
- vs raw OpenViking: "Enterprise-ready, encrypted, with governance built in"
- vs no solution: "Stop losing context. Stop leaking data."

---

## 2. Architecture

### Approach: Orchestration Layer

Context Chest defines its own API surface with business-level operations (remember, recall, connect, grant). Internally it orchestrates between OpenViking (context ops), S3 (encrypted L2 content), and Prisma (users, grants, audit). OpenViking is an implementation detail, not exposed.

```
Clients (MCP Server, Python SDK, PWA, Extension)
       │
       ▼
Context Chest API (Fastify)
  ├── OPAQUE Auth + JWT
  ├── Role Scope Gate
  ├── Rate Limiting
  ├── Audit Logging
  │
  Routes:
  ├── /v1/auth/*       (existing)
  ├── /v1/vault/*      (existing)
  ├── /v1/connect/*    (extended with roles)
  ├── /v1/memory/*     (NEW)
  └── /v1/sessions/*   (NEW)
       │
  Orchestration Layer (services/)
  ├── MemoryService
  ├── ContextService
  ├── SessionService
  ├── GrantService
  ├── StorageService
  └── UsageService
       │
       ├── OpenViking (sidecar) — L0/L1 metadata, search, sessions
       ├── S3 (MinIO/AWS) — encrypted L2 content
       └── PostgreSQL (Prisma) — users, grants, audit, URI-to-S3 mapping
```

## 3. Encryption Model

### Hybrid: metadata open, content encrypted

| Layer | Storage | Encryption | Generated by |
|---|---|---|---|
| L0 (~100 tokens) | OpenViking | Plaintext | Client |
| L1 (~2000 tokens) | OpenViking | Plaintext | Client |
| L2 (full content) | S3 | AES-GCM 256 | Client encrypts |
| Encryption key | Never leaves client | Derived via HKDF (see below) | Client |

### Key derivation and lifecycle

**OPAQUE architecture change required:** The current codebase runs OPAQUE entirely server-side (`src/plugins/opaque.ts` wraps both `OpaqueServer` and `OpaqueClient`). For zero-knowledge encryption, the OPAQUE client operations must run on the actual client device (MCP server, browser, etc.), not on the API server. The server runs only `OpaqueServer` methods. The client runs `OpaqueClient` methods locally and obtains the `export_key` — a deterministic value derived from the user's password that is stable across logins. The `export_key` is the standard OPAQUE mechanism for deriving encryption keys and never leaves the client. This is a prerequisite refactor in Phase 1.

**Key derivation flow:**

1. **Registration:** Client runs `OpaqueClient.registerInit()` and `OpaqueClient.registerFinish()` locally, obtaining the `export_key`. Client generates a random 256-bit Master Key (MK). The MK is wrapped (AES-KW or AES-GCM) using a wrapping key derived from the `export_key` via HKDF-SHA256. The wrapped MK is sent to the server via `PUT /v1/auth/master-key` and stored as `encryptedMasterKey` on the User model.
2. **Login:** Client runs `OpaqueClient.authInit()` and `OpaqueClient.authFinish()` locally, obtaining the same deterministic `export_key`. Client fetches the wrapped MK from the server (`GET /v1/auth/master-key`), derives the same wrapping key via HKDF, and unwraps the MK locally. The MK is held in memory for the session duration.
3. **Encryption:** Per-memory encryption uses AES-GCM 256 with a per-item key derived from the MK via HKDF-SHA256 (salt: memory URI, info: "context-chest-l2"). Each memory gets a unique derived key and a random 96-bit IV. The IV is prepended to the ciphertext: `encryptedL2 = base64(IV || ciphertext || auth_tag)`. Clients parse the first 12 bytes as IV when decrypting.
4. **Multi-device:** New devices log in via OPAQUE, obtain the same `export_key` (deterministic from password), derive the same wrapping key, and unwrap the same MK. No key transfer protocol needed.
5. **Key rotation:** User can generate a new MK, re-encrypt all L2 content, and replace the wrapped MK via `POST /v1/auth/master-key/rotate`. This is an expensive operation and should be offered as a settings action, not automated.

**KDF specification:**
- Algorithm: HKDF-SHA256 (RFC 5869)
- Wrapping key: HKDF(ikm=OPAQUE export_key, salt=userId bytes, info="context-chest-mk-wrap", length=32)
- Per-item key: HKDF(ikm=MK, salt=URI bytes, info="context-chest-l2", length=32)
- IV format: 96-bit random, prepended to AES-GCM ciphertext in the `encryptedL2` field

**Storage of credentials on client:**
- `~/.context-chest/credentials.json` stores: JWT (for API auth), wrapped MK (encrypted, same as server copy). It does NOT store the unwrapped MK or the OPAQUE export_key.
- If the credentials file is deleted, user re-authenticates via OPAQUE and re-derives everything.

### Security boundaries

- **Server compromise** exposes: L0/L1 summaries, email, session timestamps, directory structure, wrapped MK (useless without OPAQUE export_key, which never leaves the client). Does NOT expose: full content, unwrapped MK, passwords.
- **S3 compromise** exposes: encrypted blobs useless without user's MK.
- **Client compromise** exposes: everything for that user while MK is in memory (same threat model as 1Password/Proton).
- **L0/L1 trade-off**: Clients instructed to keep summaries generic — enough for search relevance, not enough to reconstruct sensitive content. The `browse` endpoint (min role: `tool`) exposes directory structure to any connected agent — this is an intentional trade-off for search usability.

| Permission | tool | assistant | admin |
|---|---|---|---|
| recall (search) | Yes | Yes | Yes |

### Request/Response Schemas

**POST /v1/memory/remember** request:
```json
{
  "uri": "preferences/editor-theme",
  "l0": "User prefers dark mode in code editors",
  "l1": "## Editor Preferences\n- Theme: dark mode\n- Font: monospace\n- Tab size: 2 spaces",
  "encryptedL2": "<base64 encoded AES-GCM ciphertext>",
  "sha256": "<hex hash of encrypted L2 bytes>"
}
```
`l0`: string, max 500 chars. `l1`: string, max 10,000 chars. `encryptedL2`: base64-encoded bytes, max 10MB decoded. Content-type: `application/json` (L2 is base64 in the JSON body, not multipart).

**POST /v1/memory/recall** request:
```json
{ "query": "editor preferences", "limit": 10, "offset": 0 }
```
Response follows project `ApiResponse` pattern:
```json
{ "success": true, "data": [{ "uri": "...", "l0": "...", "l1": "...", "score": 0.87 }], "meta": { "total": 42, "page": 1, "limit": 10 } }
```

**GET /v1/memory/browse** query params: `?path=preferences/&depth=2&page=1&limit=50`
Response: `{ "success": true, "data": { "tree": [{ "uri": "...", "l0": "...", "type": "file|directory", "children": [...] }] }, "meta": { "total": 15, "page": 1, "limit": 50 } }`

## 5. Database Schema

### Modified

```prisma
// Grant: scopes String[] → role Role
// Migration: add `role` column with default `tool`, backfill existing grants
//   - scopes containing "vault.write" → role = assistant
//   - scopes containing only "vault.read" → role = tool
//   - Then drop `scopes` column
// Transition: /v1/connect/token returns both `scopes` (deprecated) and `role` for one release cycle
//   - scopes derived from role: tool→["vault.read"], assistant→["vault.read","vault.write"], admin→["vault.read","vault.write"]

  assistant
  admin
}
```

### Storage Responsibilities

| Data | Storage | Why |
|---|---|---|
| User accounts, grants, audit | Prisma/Postgres | Transactional, relational |
| Memory URI to S3 key mapping | Prisma/Postgres | ACID guarantees |
| Session metadata | Prisma/Postgres | Status tracking, foreign keys |
| L0/L1 summaries + search index | OpenViking | Built for this |
| Encrypted L2 content | S3 | Blob storage |
| Session messages (encrypted) | S3 | Too large for Postgres |

## 6. Service Layer

Six services in `src/services/`:

### MemoryService
- `remember(userId, { uri, l0, l1, encryptedL2, sha256 })` — Write order: S3 upload → OpenViking write → Prisma insert. Rollback on failure in reverse order: if Prisma fails, delete from OpenViking then S3; if OpenViking fails, delete from S3.
- `recall(userId, { query, limit })` — OpenViking search over L0/L1, scoped to user namespace.
- `getContent(userId, uri)` — Prisma lookup for S3 key, fetch encrypted blob.
- `forget(userId, uri)` — Delete from OpenViking, S3, and Prisma.
- `browse(userId, { path, depth })` — OpenViking directory listing.

### ContextService
Thin wrapper around OpenViking `AsyncHTTPClient`. Only place in codebase that knows OpenViking exists. Methods: `write`, `find`, `read`, `abstract`, `delete`, `list`, `startSession`, `appendMessage`, `closeSession`.

**Namespace convention:** Each user gets an isolated root at `viking://user/{userId}/`. All ContextService methods prepend this prefix automatically. Client-supplied URIs are relative paths (e.g., `preferences/editor-theme`) that resolve to `viking://user/{userId}/memories/preferences/editor-theme`. Session data lives at `viking://user/{userId}/sessions/{sessionId}/`. This maps to OpenViking's `owner_space` isolation — each userId is a separate space.

### SessionService
- `create(userId, clientId?)` — Prisma record + OpenViking session.
- `appendMessage(userId, sessionId, { role, encryptedContent, l0Summary })` — S3 for encrypted message, OpenViking for L0, Prisma increment.
- `close(userId, sessionId, extractedMemories[])` — Client-driven memory extraction. The client (MCP server) reads back the session's L0 summaries, uses the host LLM to extract memories from the conversation, generates L0/L1 for each extracted memory, encrypts L2 content, and sends the batch to this endpoint. The server stores each memory via MemoryService, closes the OpenViking session, and updates Prisma status.

**Why client-side extraction:** The server cannot extract memories from encrypted session content without the encryption key. Memory extraction happens on the client where plaintext is available. The `close` endpoint receives pre-extracted memories, it does not trigger extraction itself. This preserves the zero-knowledge model.

**Session close flow:**
1. Client calls `GET /v1/sessions/:id` to get session L0 summaries
2. Client prompts host LLM: "Extract key memories from this conversation" using full plaintext
3. Client generates L0/L1 for each extracted memory, encrypts L2
4. Client calls `POST /v1/sessions/:id/close` with `{ memories: [{ uri, l0, l1, encryptedL2, sha256 }] }`
   - `memories` array max length: 50. Returns 400 if exceeded.
   - Total payload size still subject to existing 10MB body limit.
5. Server stores each memory via `MemoryService.remember()`, closes session in OpenViking + Prisma
6. System-initiated remembers from session close do NOT count against the user's `remember` usage quota — only explicit `POST /v1/memory/remember` calls are metered

### GrantService
- `authorize`, `exchange`, `introspect` — Extended from existing Connect flow with role-based permissions.
- `listGrants`, `revoke` — New grant management.
- `checkPermission(grant, action)` — Maps role to allowed action set.

### StorageService
Extracted from existing vault.ts: `upload`, `download`, `delete`.

### AuditService
Extracted from existing `src/plugins/audit.ts` onResponse hook. The plugin delegates to this service.
- `log(userId, { action, uri?, clientId?, ip, status, bytes? })` — Insert AuditLog record. Extends current audit with optional context-specific fields (action name, memory URI, client ID).
- The existing `audit.ts` plugin continues to fire on every response, but calls `AuditService.log()` instead of directly inserting into Prisma.

### UsageService
Increment counters per action per monthly period. Free tier enforcement via counter checks before executing operations. When limit reached, return 429 with error code `USAGE_LIMIT_REACHED`. Memory count limits (1K free, 50K starter, etc.) enforced via `COUNT(*)` query on `MemoryEntry` for the user, not via `UsageRecord` (which tracks per-period operations).

### Error Handling
Best-effort compensation for multi-backend writes. Orphaned data (encrypted blob with no Prisma reference, or metadata with no S3 content) is harmless. Periodic cleanup job reconciles later.

## 7. MCP Server

Ships as `@context-chest/mcp-server` npm package.

### Tools

| Tool | Description | Calls |
|---|---|---|
| `context-chest:remember` | Store a memory | POST /v1/memory/remember |
| `context-chest:recall` | Search memories | POST /v1/memory/recall |
| `context-chest:read` | Read full memory content | GET /v1/memory/content/:uri |
| `context-chest:forget` | Delete a memory | DELETE /v1/memory/forget/:uri |
| `context-chest:browse` | Browse memory directory | GET /v1/memory/browse |
| `context-chest:session-start` | Start tracking conversation | POST /v1/sessions |
| `context-chest:session-append` | Add a message to current session | POST /v1/sessions/:id/messages |
| `context-chest:session-save` | Extract memories and close session | POST /v1/sessions/:id/close |

### Client-side L0/L1 Generation

MCP server prompts the host LLM to generate summaries:
- L0: One-sentence abstract, no PII/secrets, topic and type only.
- L1: Structured overview with topics, concepts, entity types. No specific values.

### Auth Flow

- First-time: Browser-based OPAQUE login, JWT stored in `~/.context-chest/credentials.json`.
- Third-party agents: Connect flow with role consent prompt.
