# Context Chest v0.2 — Multi-Chest + Auto-Sort

## Overview

Users get multiple isolated chests per project. OpenViking auto-categorizes memories. Agents only see chests they're granted access to.

## User Experience

### Before (v0.1)
```
User: "Remember that Acme Corp uses PostgreSQL"
AI: Remembered at acme/database   ← user picks the path manually
```

### After (v0.2)
```
User: "Remember that Acme Corp uses PostgreSQL"
AI: Auto-sorted to chest "acme-corp" → entities/tech-stack   ← OpenViking sorts it
```

### MCP Config (per-project)
```json
{
  "mcpServers": {
    "context-chest": {
      "command": "npx",
      "args": ["context-chest-mcp", "--chest", "acme-corp"]
    }
  }
}
```

Different project → different `.mcp.json` → different chest. Zero cross-contamination.

### Access Control
```
Chest "acme-corp"     → Claude Code ✓  Cursor ✓  OpenClaw ✗
Chest "personal"      → Claude Code ✓  Cursor ✗  OpenClaw ✗
Chest "widget-inc"    → Claude Code ✓  Cursor ✓  OpenClaw ✓
Chest "global-prefs"  → all agents  ✓
```

## Architecture

### Database Schema

```prisma
model Chest {
  id          String   @id @default(uuid())
  userId      String   @map("user_id")
  name        String
  description String?
  createdAt   DateTime @default(now())
  user        User     @relation(fields: [userId], references: [id])
  memories    MemoryEntry[]
  permissions ChestPermission[]

  @@unique([userId, name])
}

model ChestPermission {
  id        String   @id @default(uuid())
  chestId   String   @map("chest_id")
  agentName String   @map("agent_name")
  canRead   Boolean  @default(true)
  canWrite  Boolean  @default(true)
  chest     Chest    @relation(fields: [chestId], references: [id])

  @@unique([chestId, agentName])
}
```

MemoryEntry gets a `chestId` foreign key. Default chest = "default" (backwards compatible).

### Encryption

Per-chest key derivation:
```
Master Key
  └─ HKDF(masterKey, chestName + memoryURI)
     └─ per-item AES-256-GCM key
```

Different chest = different key derivation salt = different ciphertext even for identical content. Compromising one chest's keys reveals nothing about another.

### API Changes

```
POST   /v1/chests                    Create a chest
GET    /v1/chests                    List user's chests
PUT    /v1/chests/:id/permissions    Set agent access
DELETE /v1/chests/:id                Delete a chest

# Existing endpoints gain ?chest= query param:
POST   /v1/memory/remember?chest=acme-corp
POST   /v1/memory/recall?chest=acme-corp
GET    /v1/memory/browse?chest=acme-corp
```

### MCP Server Changes

```
context-chest-mcp --chest acme-corp
```

The `--chest` flag is passed via MCP config args. The MCP server includes `X-Chest: acme-corp` header on all API calls. If no chest specified, uses "default".

### OpenViking Auto-Sort

When a user stores a memory without a path:

1. Content sent to OpenViking for analysis
2. OpenViking extracts category (profile/preferences/entities/events/cases/patterns)
3. OpenViking suggests directory path based on semantic similarity to existing memories
4. Memory stored at auto-generated path within the active chest

```
User: "Remember this"
  ↓
OpenViking session.commit()
  ↓
Extracts: category=entity, topic=database, entity=Acme Corp
  ↓
Auto-path: entities/acme-corp/tech-stack
  ↓
Encrypted + stored in chest "acme-corp"
```
