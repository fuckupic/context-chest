# Context Chest v0.3 — Enterprise Teams + Privacy Guard

## Status: ROADMAP (not started)
Build when first enterprise customer reaches out.

---

## Overview

Teams share chests across members while keeping personal data private. Personal chests NEVER become shared. Team chests are created explicitly by admins. No automatic merging — "share to team" is always an explicit action.

## Core Principle: No Merge, No Automatic Sharing

```
RULE 1: Personal chests stay personal. Always.
RULE 2: Team chests are new, empty, explicitly created.
RULE 3: Auto-chest NEVER routes to team chests.
RULE 4: "Share to team" copies a memory — it doesn't move it.
```

---

## How It Works — The Bluffish Example

### Before teams:
```
Tady's vault:
  work (personal)     → 20 Bluffish memories
  health (personal)   → dentist, gym
  finance (personal)  → burn rate, pricing
```

### Tady creates a team chest and invites Pavel:
```
Tady's vault:
  work (personal)     → 20 Bluffish memories (unchanged, only Tady sees)
  health (personal)   → dentist, gym (only Tady)
  finance (personal)  → burn rate (only Tady)
  bluffish-team (shared) → empty, new

Pavel's vault:
  work (personal)     → his own notes (only Pavel sees)
  bluffish-team (shared) → same shared chest as Tady
```

### From now on:
- New Bluffish context from EITHER user goes to `bluffish-team` (when in team mode)
- Personal auto-chest still routes to personal chests (health, finance, etc.)
- Old personal memories stay personal — Tady can "share to team" specific ones

### Sharing existing memories:
```
Tady clicks "SHARE TO TEAM" on a personal memory
  → Memory is COPIED to bluffish-team chest
  → Original stays in personal work chest
  → Re-encrypted with team key
  → Pavel can now see it
```

## Team Model

### Organization → Members
```
Organization (e.g., "Uhumdrum s.r.o.")
  └─ Members with roles (owner, admin, member, viewer)
  └─ Team chests (bluffish-team, ops-runbooks, etc.)
```

### Roles
- **Owner** — full control, billing, member management, create/delete team chests
- **Admin** — manage team chests, invite members
- **Member** — read/write to team chests they have access to
- **Viewer** — read-only access to team chests

---

## Chest Types

### Personal Chests (existing, unchanged)
- Owned by one user, only that user can access
- Auto-created categories (work, health, finance, personal, learning, tools) are ALWAYS personal
- Auto-chest routing ALWAYS goes to personal chests
- Cannot be converted to team chests

### Team Chests (new)
- Owned by an organization
- Created explicitly by owner/admin — never auto-created
- All org members can access (or scoped by permission)
- Examples: `bluffish-team`, `engineering-decisions`, `product-roadmap`

### Rules
- `health`, `personal`, `finance` categories can NEVER be team chests
- Auto-chest routing NEVER creates or routes to team chests
- Team chests have a distinct visual indicator (team icon, different border)
- MCP server needs explicit `--team=bluffish-team` flag or env var to write to team chest

## "Share to Team" Flow

### In PWA:
1. User views a personal memory in the editor
2. Clicks "SHARE TO TEAM" button
3. Selects which team chest to share to
4. Memory is **copied** (not moved) to team chest
5. Content is decrypted with personal key, re-encrypted with team key
6. Original personal memory is untouched

### Via MCP:
```
context-chest_remember(content: "...", chest: "bluffish-team")
```
Explicit chest parameter = writes to team chest. No parameter = personal auto-routing.

### What "share" means technically:
1. Decrypt memory with personal master key
2. Re-encrypt with team master key
3. Create new memory entry in team chest
4. New URI, new encryption, new S3 key
5. Original personal memory unchanged

## Encryption for Teams

### Per-Team Key Wrapping
```
Team Master Key (256-bit random, created when team chest is created)
  │
  ├─ Wrapped for Tady:  AES-wrap(tady.masterKey, teamKey)
  ├─ Wrapped for Pavel:  AES-wrap(pavel.masterKey, teamKey)
  └─ Wrapped for Luky:  AES-wrap(luky.masterKey, teamKey)
```

- Each team chest has its own master key
- Key is wrapped separately for each member using their personal master key
- Member opens team chest → unwraps team key with their personal key → decrypts memories

### When a member is removed:
1. Generate NEW team master key
2. Re-encrypt ALL memories in that team chest
3. Re-wrap new key for remaining members only
4. Removed member's wrapped key is deleted — they can no longer decrypt

### When a member is added:
1. Wrap existing team key for new member using their personal key
2. New member can now decrypt all team chest memories

## Auto-Chest Behavior with Teams

### Personal mode (default):
- Auto-chest routes to personal chests as usual
- Team chests are INVISIBLE to auto-routing
- "bluffish" keywords → personal work chest

### Team mode (explicit):
- MCP config: `"CONTEXT_CHEST_TEAM": "bluffish-team"` in env
- Or per-request: `context-chest_remember(content: "...", chest: "bluffish-team")`
- Only explicit writes go to team chests

### Why no automatic team routing:
- Risk of personal data leaking into shared space
- "I mentioned my salary while discussing Bluffish" → must NOT go to team chest
- Explicit is safe. Automatic is risky.

## Database Schema (Draft)

```prisma
model Organization {
  id        String      @id @default(uuid())
  name      String
  createdAt DateTime    @default(now())
  members   OrgMember[]

  @@map("organizations")
}

model OrgMember {
  id     String  @id @default(uuid())
  orgId  String  @map("org_id")
  userId String  @map("user_id")
  role   OrgRole // owner, admin, member, viewer
  org    Organization @relation(fields: [orgId], references: [id])
  user   User         @relation(fields: [userId], references: [id])

  @@unique([orgId, userId])
  @@map("org_members")
}

enum OrgRole {
  owner
  admin
  member
  viewer
}

// Chest gains optional orgId
model Chest {
  // existing fields...
  orgId  String?  @map("org_id")  // null = personal, set = team chest
  org    Organization? @relation(...)

  // Team key (wrapped per-member, stored as JSON array)
  teamKeyWraps  Json?  @map("team_key_wraps")
}
```

## API Changes

### New Endpoints
```
POST   /v1/orgs                          Create organization
GET    /v1/orgs                          List user's orgs
POST   /v1/orgs/:id/members             Invite member (by email)
DELETE /v1/orgs/:id/members/:userId      Remove member
PUT    /v1/orgs/:id/members/:userId      Change role
POST   /v1/orgs/:id/chests              Create team chest
DELETE /v1/orgs/:id/chests/:chestId     Delete team chest
POST   /v1/memory/share                 Copy personal memory to team chest
```

### Modified Endpoints
- `GET /v1/chests` — returns personal + team chests (with `orgId` indicator)
- `POST /v1/memory/remember` — accepts `chest` param for explicit team chest writes
