## Overview

Teams share chests across members while keeping personal data private. Privacy Guard ensures sensitive content (health, family, salary) never leaks into shared chests. Per-member key wrapping maintains E2E encryption even for team content.

## Team Model

### Org ‚Üí Team ‚Üí Members
```
Organization (company)
  ‚îî‚îÄ Teams (engineering, design, ops)
       ‚îî‚îÄ Members (users with roles)
```

### Roles
- **Owner** ‚Äî full control, billing, member management
- **Admin** ‚Äî manage chests, permissions, members
- **Member** ‚Äî read/write to team chests they have access to
- **Viewer** ‚Äî read-only access to team chests

## Chest Types

### Personal Chests (existing)
- Owned by one user
- Only that user can access
- Never visible to team members
- Auto-created categories (work, health, finance, personal, learning, tools) stay personal by default

### Team Chests (new)
- Owned by an organization
- Visible to all team members (or scoped by team)
- Created by admins
- Examples: `team/engineering/architecture`, `team/ops/runbooks`, `team/product/decisions`

### Rules
- `health`, `personal`, `finance` chests are ALWAYS personal ‚Äî cannot be converted to team
- `work` chest can be personal or team (user choice)
- New team chests are explicitly created ‚Äî auto-sort never creates team chests automatically

## Privacy Guard

### Auto-Classification
When auto-chest routes a memory, Privacy Guard checks:

1. Does the content contain sensitive keywords? (salary, health, family, personal, password, SSN, etc.)
2. Is the target chest a team chest?
3. If both ‚Üí redirect to personal chest instead

### Sensitivity Keywords (configurable per org)
```
HIGH SENSITIVITY (always personal):
  health, doctor, dentist, medication, diagnosis, therapy
  salary, compensation, bonus, equity, stock options
  family, wife, husband, kids, divorce, personal
  password, secret key, private key, SSN, social security

MEDIUM SENSITIVITY (warn before sharing):
  revenue, burn rate, runway, fundraising
  hiring, firing, performance review
  legal, lawsuit, NDA terms
```

### Admin Controls
- Org admin sets sensitivity rules
- Can add custom keywords per org
- Can set chests as "personal-only" (never shareable)
- Audit log shows when Privacy Guard blocked a share

## Encryption for Teams

### Challenge
Current model: each user has their own master key. Team chests need all members to decrypt.

### Solution: Per-Team Key Wrapping
```
Team Master Key (256-bit random, created when team is created)
  ‚îÇ
  ‚îú‚îÄ Wrapped for User A: HKDF(userA.masterKey, teamId) ‚Üí AES wrap
  ‚îú‚îÄ Wrapped for User B: HKDF(userB.masterKey, teamId) ‚Üí AES wrap
  ‚îî‚îÄ Wrapped for User C: HKDF(userC.masterKey, teamId) ‚Üí AES wrap
```

- Each team has its own master key
- That key is wrapped (encrypted) separately for each member using their personal master key
- When a member opens a team chest, they unwrap the team key using their personal key
- When a member is removed: generate new team key, re-wrap for remaining members, re-encrypt team chests

### Key Rotation on Member Removal
1. Generate new team master key
2. Re-encrypt all team chest memories with new key
3. Wrap new key for each remaining member
4. Old key is destroyed ‚Äî removed member can no longer decrypt

## API Changes

### New Endpoints
```
POST   /v1/orgs                        Create organization

POST   /v1/orgs/:id/teams              Create team

```

### Modified Endpoints
- `POST /v1/memory/remember` ‚Äî Privacy Guard check before storing in team chest
- `GET /v1/chests` ‚Äî returns both personal and team chests (with type indicator)
- `POST /v1/memory/auto-chest` ‚Äî skips team chests, only routes to personal

## Database Schema (Draft)

```prisma
model Organization {
  id        String   @id @default(uuid())
  name      String
  createdAt DateTime @default(now())
  teams     Team[]
  members   OrgMember[]
  chests    Chest[]        // team chests owned by org
}

model OrgMember {
  id     String @id @default(uuid())
  orgId  String
  userId String
  role   OrgRole // owner, admin, member, viewer
  org    Organization @relation(...)
  user   User @relation(...)
  @@unique([orgId, userId])
}

model Team {
  id    String @id @default(uuid())
  orgId String
  name  String
  org   Organization @relation(...)
  @@unique([orgId, name])
}

// Chest gains optional orgId
model Chest {
  orgId  String?  // null = personal, set = team chest
  org    Organization? @relation(...)
}
```
